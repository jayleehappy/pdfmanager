## 一、项目开发目标

本工具是一款基于 Python + PySide6 的 Windows 桌面应用，用于：

1. **目录扫描与树形展示** — 扫描指定工作目录，显示目录和 PDF 文件的树形结构
2. **树结构编辑** — 支持拖拽排序、重命名、删除节点等操作
3. **PDF 预览** — 集成 PDF 渲染，支持鼠标滚轮缩放（50%~300%）
4. **XML 标签导入/导出** — 按照 PDF 书签标准格式读写 XML，与文件结构互相校验
5. **PDF 合并** — 按编辑后的树结构顺序合并多个 PDF，并生成与目录层级对应的嵌套书签
6. **PDF 拆分（今日重点）** — 按书签层级拆分为目录结构和独立 PDF 文件，是合并的逆过程

## 四、今日实现的功能

### 4.1 PDF 拆分引擎完全重写

**目标**：实现合并的逆过程——读取 PDF 书签，按书签层级生成目录结构，叶子节点提取为独立 PDF。

**最终实现逻辑**：

- **书签结构分析**：pypdf 生成的 `merged.pdf` 使用扁平列表+内嵌子列表的混合结构：
  ```
  outline = [DEST(目录, Count>0), LIST(子节点), DEST, LIST, ..., DEST(叶子, Count=0)]
  LIST 内部：交替出现 DEST 和嵌套 LIST
  ```
- **树节点类型**：`is_leaf()` 基于 `children` 而非 `page_index`（因为目录书签 `Count>0` 也有 `page_index`）
- **页码解析**：`item.page` 返回 `IndirectObject`，通过 `indirect_reference` 属性逐一与 `reader.pages` 比较才能匹配
- **范围分配**：叶子节点页码范围 `[start, end)`（左闭右开），通过递归修正兄弟叶子间的边界

**测试结果**（以 `output/merged.pdf` 为例）：
- 原 PDF：18 页，3 级嵌套书签，18 个叶子节点
- 拆分结果：18 个独立 PDF 文件（各 1 页），目录结构与书签层级完全一致


## 五、遇到的问题及解决方法

### 问题 1：书签遍历方式错误
- **现象**：`解析书签项失败: 'method' object is not iterable`
- **根因**：书签项的 `children` 是方法（`bound method`），不是可迭代对象；且 pypdf 的 outline 结构为扁平列表与内嵌子列表的混合，不能简单遍历
- **解决**：改用递归下降解析器，手动解析 `[DEST, LIST, DEST, LIST, ...]` 交替结构

### 问题 2：页码解析返回 None
- **现象**：所有叶子节点 `page_index=None`，导致页码范围全是 `0-0` 或 `0-1`
- **根因**：`item.page` 返回 `IndirectObject`，与 `reader.pages` 中的 `PageObject` 直接 `==` 比较永远为 `False`
- **解决**：用 `indirect_reference` 属性进行比较（该属性在两边都存在且值相同）

### 问题 3：`is_leaf()` 判断逻辑错误
- **现象**：目录书签（`Count>0`）因有 `page_index` 被误判为叶子；叶子书签因 `page_index=None`（初始未解析）被误判为目录
- **根因**：`is_leaf()` 基于 `page_index is not None`，但 `Count>0` 的目录书签也有 `page_index`（指向该节首页）
- **解决**：将 `is_leaf()` 改为基于 `children`（即 `len(children) == 0`）；在构建树时直接通过 `raw_item` 解析页码，不再依赖延迟解析

### 问题 4：页码范围边界传播错误
- **现象**：嵌套目录中内层叶子范围出现 `pages=12-6`（start > end）
- **根因**：`_assign_page_ranges` 递归修正时，父目录范围的计算结果被子层修正覆盖，导致边界相互干扰
- **解决**：简化为两步：① 递归计算所有节点初始范围；② 在当前层直接修正叶子兄弟边界（`curr.end = next.start`），不再向上回溯覆盖父目录范围

### 问题 5：提取的 PDF 文件无扩展名
- **现象**：`PermissionError: ... 未命名书签`（目录当文件写入时无写权限）
- **根因**：叶子节点目录路径与 PDF 文件路径混淆；叶子节点提取时路径未加 `.pdf` 扩展名
- **解决**：叶子节点统一输出为 `{safe_name}.pdf`，目录节点只建目录不建文件

## 六、核心数据结构

### OutlineNode（pdf_splitter.py）
```python
class OutlineNode:
    title: str                    # 显示名称
    page_index: Optional[int]     # 书签指向的 0-based 页码
    start_page: Optional[int]     # 提取起始页（含）
    end_page: Optional[int]      # 提取结束页（不含）
    children: List[OutlineNode]  # 子节点列表
    parent: Optional[OutlineNode] # 父节点引用
    raw_item                     # 原始 pypdf Destination 对象
```

```python
    children: List[TreeItem]     # 子节点
    parent: Optional[TreeItem]   # 父节点引用
```

## 七、pypdf 书签结构参考

`merged.pdf` 的书签是 pypdf 合并时自动生成的扁平结构：
- 顶层交替：`[DEST, LIST, DEST, LIST, ...]`
- `DEST`：书签节点，`Count>0`=目录，`Count=0`=叶子
- `LIST`：子节点列表，内部可能包含 `DEST` 和嵌套 `LIST`
- `Count` 属性：表示该书签下有多少个子书签（用于 PDF 阅读器展开）

**关键属性**：
- `item.title` — 书签标题
- `item.page` — 指向的页面（`IndirectObject` 类型，需用 `indirect_reference` 比对）
- `item.get("/Count", 0)` — 子书签数量
- `item.children()` — 子书签生成器（需 `list()` 转换）

### 9.2 今日遇到的问题及解决

#### 问题 1：onefile 模式"选择工作目录"按钮无反应
- **现象**：点击按钮后控制台无输出，目录选择窗口不出现；其他按钮（导入标签、拆分PDF）正常
- **根因**：onefile 模式下，`getExistingDirectory` 使用 `SHBrowseForFolder`（Windows Shell COM 组件），
  而 PyInstaller 引导程序在 Temp 目录解压时，COM 子系统初始化不完整，导致 Shell 命名空间扩展失败
- **对比**：`getOpenFileName`（文件选择）不依赖这套 COM，所以正常
- **解决**：改用 onedir 模式（目录分发），稳定性远高于 onefile

#### 问题 2：pypdfium2 报错"需要安装 pypdfium2"
- **现象**：点击 PDF 文件后预览区提示"需要安装 pypdfium2"
- **诊断**：控制台显示 `ImportError: No module named 'PIL'`
- **根因**：打包后的 PIL 没有被包含，导致 `page.render().to_pil()` 失败
  ```
  page.render().to_pil()  ← 需要 Pillow（PIL）
  except ImportError       ← 被误报为 pypdfium2 未安装
  ```
- **解决**：从 spec 的 `excludes` 中移除 `"PIL"`，并在 `hiddenimports` 中添加 `"PIL"`、`"PIL.Image"`

#### 问题 3：PDF 拆分文件名双后缀
- **现象**：拆分出的文件名如 `03-王五五五.pdf.pdf`
- **根因**：书签标题本身已含 `.pdf` 后缀（如"03-王五五五.pdf"），`_extract_leaves` 又拼接了 `.pdf`
- **解决**：提取文件名时检测是否已含 `.pdf`，已含则不再追加

#### 问题 4：导入标签报错 `AttributeError`
- **现象**：`tree_manager.items_to_tree_items` 方法不存在
- **根因**：该方法定义在 `XMLHandler` 类中，`TreeManager` 没有此方法
- **解决**：在 `xml_handler.py` 中补了模块级 `items_to_tree_items` 便捷函数

#### 问题 5：onedir 模式下 PySide6 插件路径错误
- **现象**：Qt 插件（platforms、imageformats）在 `_internal/` 子目录下，但 `QT_PLUGIN_PATH` 指向错误路径
- **诊断**：`sys._MEIPASS` 在 onedir 模式下等于 `_internal/` 目录
- **解决**：rthook.py 中判断 onedir 模式（`_MEIPASS` 末尾为 `_internal`），正确计算 exe 所在目录

#### 问题 6：pypdfium2 包不完整
- **现象**：bundled 的 `pypdfium2/` 只有 `version.json`，Python 包文件缺失
- **根因**：spec 中的 `datas` 只指定了目录名，未正确包含完整包
- **解决**：在 spec 中显式添加 `datas.append((str(pypdfium2_pkg_dir), "pypdfium2"))` 和
  `datas.append((str(pypdfium2_raw_pkg_dir), "pypdfium2_raw"))`

#### 问题 7：pypdfium2_raw 包缺失
- **现象**：bundled 的 `pypdfium2_raw/` 只有 `pdfium.dll` 和 `version.json`，缺少 `bindings.py`
- **根因**：`pypdfium2_raw` 是 site-packages 的顶级包（不是子包），PyInstaller 未自动打包
- **导入链**：`pypdfium2.raw → from pypdfium2_raw.bindings import *`
- **解决**：在 spec 中通过 `import pypdfium2_raw` 获取其路径，显式打包完整包

#### 问题 8：sys.path 优先级导致包导入错误
- **现象**：Python 优先从系统 site-packages 导入 `pypdfium2_raw`（缺 bindings.py），
  而非 bundled 的 `_internal/pypdfium2_raw`（完整）
- **根因**：PyInstaller 引导程序将 `_internal` 加到 `sys.path` 时位置不够靠前
- **解决**：在 rthook.py 最开头强制将 `_internal` 插入 `sys.path[0]`

#### 问题 9：vcruntime140_1.dll 不在 System32
- **现象**：PyInstaller onefile 模式报错 `failed to load python dll`
- **根因**：`vcruntime140_1.dll` 只存在于 Python 安装目录，不在 `C:\Windows\System32\`
- **解决**：在 spec 的 `binaries` 中显式添加 `vcruntime140.dll` 和 `vcruntime140_1.dll`

- 合并时虚拟书签自动插入一页 A4 空白页占位，目录书签正常生成

#### 5. 添加 PDF 文件（文件选择框）
- 右键菜单「添加 PDF 文件」改为弹出系统文件选择框

- 选择后以文件名（去 `.pdf` 后缀）作为显示名，`path` 存入完整路径

#### 6. 缺失文件预检 + 空白页自动生成
- 合并前预遍历树，收集**文件不存在**的真实文件节点（XML 导入的 `path=""` 节点）
- 弹窗列出缺失节点名称，用户确认后：

### 遇到的问题及解决方法

#### 问题 1：f-string 中文引号语法错误
- **现象**：`SyntaxError: invalid syntax`，位置在 `main_window.py:308`
- **根因**：f-string 内 `\"` 被 Python 解释器当作转义字符
- **解决**：改用 Unicode 转义 `"\u201c"` / `"\u201d"` 替代中文引号字符

#### 问题 2：排序键比较 TypeError
- **现象**：`TypeError: '<' not supported between instances of 'int' and 'str'`
- **根因**：`key_parts` 列表中数字段解析为 `int`，其余字符直接放入列表（如 `'a'`），比较 `97 < 't'` 报错
- **解决**：非数字段字符改用 `key_parts.append(ord(rest[0]))`，保证全为整数比较

#### 问题 3：INI 排序优先级反转
- **现象**：INI 中定义的项目反而排在未定义项目之后
- **根因**：INI 项返回 `(0, [])`，未定义项返回 `(-1, ...)`；Python 元组比较 `(-1, x) < (0, y)`
- **解决**：INI 项改为 `(-1 - n, [])`（0→-1, 1→-2...），保证始终小于未定义项的 `(-1, ...)`

#### 问题 4：XML 导入后选择节点预览报错
- **现象**：`main_window._scan_and_load_directory` 末尾调用了不存在的方法 `_load_xml_into_tree`
- **解决**：新增 `_on_preview_request(tree_item)` 处理器，信号改为传递 `TreeItem` 对象以保留完整 parent 链

#### 问题 5：XML 导入时预览路径错误
- **现象**：预览提示"文件不存在"，实际文件在深层子目录中
- **根因**：`path=""` 时只用文件名拼 `work_dir/文件名.pdf`，丢失了父目录链
- **解决**：从 `TreeItem.parent` 链向上回溯，收集所有目录名反序拼接完整相对路径

#### 问题 6：XML 节点验证忽略嵌套目录
- **现象**：加载 XML 后节点验证报告所有节点"在工作目录中未找到"
- **根因**：`_validate_item` 中 `alt_path = Path(work_dir) / name` 只拼了文件名，没有补上递归累积的 `parent_path`
- **解决**：`alt_path = (Path(work_dir) / parent_path / name)`，同时对非 `.pdf` 后缀的 name 兜底检查 `name + '.pdf'`

#### 问题 7：合并时"没有找到可合并的 PDF"
- **现象**：树中有文件节点，但合并时提示无文件
- **根因**：`_collect_pdf_files` 只看 `item.path`，XML 导入的节点 `path=""` 被跳过
- **解决**：统一使用 `TreeItem.resolve_path(work_dir)`，path 为空时用 parent 链重建

#### 问题 8：虚拟书签合并后消失
- **现象**：添加的虚拟书签在合并 PDF 中没有空白页，也没有书签
- **根因**：`_process_tree_item` 判断 `if not resolved`（resolved 是构造路径非空），走了 `_add_pdf_with_bookmark`，但文件不存在打印警告后 `return`
- **解决**：改为按 `item.path` 是否为空判断：`path=""` → `_add_blank_page_with_bookmark`；`path≠""` 且文件存在 → `_add_pdf_with_bookmark`

#### 问题 9：合并后全部页面为空白页（最严重 Bug）
- **现象**：选择合并后弹出缺失列表，确认后生成合并 PDF，全部页面空白
- **根因（多层）**：
  1. `_collect_missing_nodes` 收集了**所有文件节点**，包括真实存在的 PDF
  2. 预生成空白页时用 `m._write_blank_pdf(resolved_path)` **覆盖了真实 PDF 文件**
  3. Windows 下 `Path('D:/work/...').exists()` 对含正斜杠的路径返回 `False`，导致所有文件都判断为"不存在"
  4. 合并时 `_add_pdf_with_bookmark` 读取的正是被覆盖后的空白文件
- **解决**：
  - `tree_manager.resolve_path`：改用 `os.path.join()` 确保路径用反斜杠（跨平台）
  - `_collect_missing_nodes`：`resolved and not os.path.exists(os.path.normpath(resolved))` — 仅收集真正不存在的节点
  - `_add_pdf_with_bookmark`：预生成路径在 `_pregenerated_paths` 中时跳过读取，直接追加空白页（内容已在预检时写入磁盘）
  - `xml_handler.py` 验证逻辑同样改用 `os.path` 方法替代 `Path.exists()`

### 核心设计更新

#### 虚拟节点路径解析（最重要）
```
TreeItem.resolve_path(work_dir):
  1. path 非空 → 直接返回（真实文件）
  2. path 为空 → 沿 parent 链向上收集目录名 → 反序拼接 → 末尾补 .pdf → os.path.join
```
此方法是所有路径相关逻辑的核心，所有模块（merger、validator、preview）共享。

#### 合并流程（最终版）
```
用户点击合并
  → get_current_structure() 重建树（最新 UI 状态）
  → _collect_missing_nodes() 收集文件不存在的节点
      → 有缺失 → 弹窗确认 → 生成空白 PDF + 记录 pre_paths
  → merge_pdfs(items, output, work_dir, callback, pre_paths)
      → item.is_dir → 创建目录书签，递归子节点
      → item.path ≠ "" → _add_pdf_with_bookmark（存在则加页；pre_paths 中则追加空白页）
      → item.path = "" → _add_blank_page_with_bookmark（直接追加空白页 + 书签）
```

#### 双树同步
```
扫描目录 / 导入 XML / 添加节点
  → TreeManager.load_from_nodes() / load_from_dict()
  → TreeWidget.load_data() — TreeItem → FileTreeItem（Qt）

用户编辑（增删改、重排序）
  → structure_changed 信号
  → TreeWidget.get_current_structure() — 从 QTreeWidget 重建 TreeItem 树
  → TreeManager.root_items 同步

预览 / 合并 / 导出
  → 统一使用 get_current_structure() 获取最新树结构
```

### 经验总结

1. **Windows 路径兼容性**：`pathlib.Path` 在 Windows 下对正斜杠字符串（`"D:/work/..."`）的 `.exists()` 返回 `False`，必须用 `os.path.join()` 或 `os.path.normpath()` + `os.path.exists()` 代替。所有涉及文件存在性判断的代码都应使用 `os.path` 而非 `pathlib.Path`。

2. **预生成文件去重**：在合并前预生成文件时，必须追踪已生成的路径集合（`_pregenerated_paths`），防止后续合并逻辑再次写入。

3. **判断条件语义明确**：`item.path`（原始属性）和 `resolve_path()`（计算属性）含义不同，必须区分使用。真实文件节点的 `path` 非空，虚拟书签的 `path` 为空。

4. **测试数据隔离**：每次测试前清理残留文件（`D:/work` 目录），避免残留的空白页文件干扰测试结果。

5. **Qt 信号类型设计**：传递 `TreeItem` 对象而非 `path` 字符串，保留完整的 parent 链信息，避免多层传递导致数据丢失。
