## 一、项目需求

### 1.1 业务背景

本系统用于辅助核查领导干部个人有关事项报告表。实际工作流程如下：

```
组织部门下发空白模板（电子版）→ 领导干部填写后扫描为 PDF → 上传系统
        ↓
系统识别手写内容 → 与反馈数据（房产/保险/护照等）自动比对
        ↓
输出不一致项清单 → 组织部门人工核查
```

### 1.2 报告表结构

报告表共 **22 页**，涵盖以下类别：

| 类别 | 页码 | 说明 |
|------|------|------|
| 基本信息 | 5 | 姓名、身份证、工作单位等 |
| 婚姻状况 | 5 | 婚姻现状及变化 |
| 健康状况 | 6 | 重大疾病情况 |
| 护照及出国 | 6-7 | 因私出国、港澳台通行证 |
| 配偶情况 | 7-9 | 配偶基本信息 |
| 子女从业 | 7-9 | 子女工作单位、职务（支持多人） |
| 移居境外 | 9-10 | 配偶子女移居情况 |
| 工资收入 | 11 | 工资、奖金、津贴 |
| 劳务所得 | 11 | 讲学、写作、咨询等 |
| 房产 | 11-13 | 境内房产（20+字段）、境外房产 |
| 股票基金 | 13-16 | 境内/境外股票、基金 |
| 投资型保险 | 16-18 | 境内/境外投资型保险 |
| 投资企业 | 18-19 | 配偶子女开办或投资的企业 |
| 中介机构 | 19-20 | 配偶子女开办有偿社会中介机构 |
| 私募股权基金 | 20-21 | 投资私募基金详情 |
| 境外存款 | 21 | 境外银行账户 |
| 其他事项 | 22 | 其他需报告内容 |

详细字段定义见：[报告表字段清单.md](../报告表字段清单.md)

### 1.3 核心需求

1. **PDF 文字识别（OCR）**
   - 将扫描件 PDF 中的文字、表格提取为结构化文本
   - 支持表格识别（HTML 格式输出）
   - 表格内容应保持行列结构

2. **模板对比优化**
   - 报告表有大量固定格式（表格线、说明文字）
   - 通过模板对比，只识别手写/填写的区域
   - 避免重复识别模板已有内容

3. **与反馈数据比对**
   - 支持导入组织部门反馈数据（Excel）
   - 将 OCR 识别结果与反馈数据进行字段级比对
   - 标注不一致项

4. **报告生成**
   - 导出比对结果为结构化文档
   - 支持打印或电子归档

## 二、技术选型

### 2.1 OCR 引擎选型

**备选方案对比：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| Tesseract OCR | 开源免费、部署简单 | 对中文识别精度低、表格识别差 |
| 百度 OCR API | 中文识别精度高 | 需要外网、收费、有数据安全顾虑 |
| 腾讯云 OCR | 中文识别精度高 | 需要外网、收费 |
| PaddleOCR VL | 开源、中文优化好、支持表格 | 需本地部署、CPU 推理慢 |
| PaddleOCR v5（PP-OCRv5） | 速度快、精度高 | 不支持表格结构解析 |

**最终选择：PaddleOCR VL 1.5**

理由：
- 开源自托管，满足数据安全要求（报告表含个人隐私）
- 内置版面分析（Layout Detection），可识别文本/表格/图片区块
- 表格识别直接输出 HTML 结构
- `pipeline.predict()` API 支持批量处理

**不选 PaddleOCR v5 的原因：**
v5 的 PP-OCR 管道不包含表格结构解析，需配合 DocParser 使用，集成复杂度高。

### 2.2 框架选型

**后端框架：FastAPI**
- 异步支持好，适合 I/O 密集型任务
- 自动生成 OpenAPI 文档
- Pydantic 数据校验

**PDF 处理：PyMuPDF（fitz）**
- 纯 Python 实现，依赖少
- PDF 转图像速度快
- 支持精确页面操作

**图像处理：PIL + NumPy**
- 灰度转换、resize 等基础操作
- 像素级差分计算

**连通域检测：SciPy**
- `scipy.ndimage.label` 实现连通域分析
- 找出差异区域边界框

### 2.3 技术栈汇总

```
Python 3.10+
├── PaddlePaddle 3.3.1 (CPU)     — 深度学习框架
├── PaddleOCR VL 1.5             — OCR 引擎
├── FastAPI + Uvicorn            — Web 框架
├── PyMuPDF (fitz)               — PDF 处理
├── PIL / Pillow                  — 图像处理
├── NumPy                         — 数值计算
├── SciPy                         — 科学计算（连通域）
├── Pydantic                     — 数据校验
└── (可选) Redis                  — 任务队列（生产环境）
```

## 三、系统架构

```
客户端 (HTML/JS)
    ↓ HTTP
FastAPI 服务端 (server.py)
    │
    ├─ api/upload.py      → PDF / Excel 文件上传
    ├─ api/ocr.py         → OCR 任务调度（异步 + 后台线程）
    ├─ api/compare.py      → 与反馈数据比对
    └─ api/report.py       → 报告生成
    │
    └─ services/
        ├─ ocr_service.py         → PaddleOCRVL 封装（单例）
        └─ template_service.py    → 模板对比（图像差分 + 连通域）
```

### 核心模块职责

| 模块 | 职责 | 关键设计 |
|------|------|---------|
| `ocr_service.py` | PaddleOCRVL 引擎封装 | 单例模式，CPU 优化参数，`_ensure_native()` 处理 numpy 类型 |
| `template_service.py` | 模板对比 | `_normalize_image_size()` 处理不同 DPI，`compute_diff_mask()` 逐像素差分，`find_diff_regions()` 连通域检测 |
| `api/ocr.py` | OCR 任务管理 | 后台线程执行，内存任务队列，长轮询查询进度 |

## 四、已实现的功能

### 4.1 PDF 上传与存储
- 上传 PDF 文件，自动分配 UUID
- 保存到 `uploads/pdf/{uuid}.pdf`
- 支持前端页面直接拖拽上传

### 4.2 OCR 识别（两种模式）

**普通模式：**
- 直接对整个 PDF 进行 OCR
- 逐页处理，输出每页的文本和表格

**模板对比模式（核心优化）：**
```
1. 将模板 PDF 和扫描件 PDF 转为灰度图像
2. 归一化尺寸（处理不同 DPI 扫描件）
3. 逐像素差分 → 二值差异遮罩
4. scipy.ndimage.label 连通域分析
5. 提取差异区域边界框
6. 仅对差异区域调用 OCR
7. 合并输出
```

### 4.3 任务队列与进度查询
- 任务异步执行（后台线程）
- 实时进度（0-100%）
- 结果持久化到 JSON 文件

### 4.4 API 接口

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/upload/pdf` | POST | 上传 PDF 文件 |
| `/api/ocr/recognize` | POST | 提交 OCR 任务 |
| `/api/ocr/status/{task_id}` | GET | 查询任务进度 |
| `/api/ocr/result/{task_id}` | GET | 获取识别结果 |

## 五、遇到的问题及解决方案

### 5.1 PaddlePaddle 环境配置

**问题：虚拟环境兼容性问题**

早期尝试在 `.venv_paddleocr` 虚拟环境中安装 PaddlePaddle 3.3.1，但遇到 MKLDNN 初始化失败、模型加载异常等问题。

**原因分析：**
- PaddlePaddle 的 CPU 加速依赖 MKL-DNN（Intel Math Kernel Library）
- Windows 虚拟环境中的 DLL 加载路径与系统 Python 不同
- `.venv` 隔离了系统 site-packages，导致依赖缺失

**解决方案：**
- **始终使用系统默认 Python 环境**，不要用虚拟环境
- 如已在虚拟环境中安装，需删除重建：`rm -rf .venv_paddleocr`
- 安装命令：
  ```powershell
  python -m pip install paddlepaddle==3.3.1 -i https://www.paddlepaddle.org.cn/packages/stable/cpu/
  ```

**核心规则（绝对禁止）：**
> **PaddlePaddle 绝对禁止降级！**
> 如果 PaddleOCR 有兼容性问题，优先尝试旧版 PaddleOCR，而不是降级 PaddlePaddle。

### 5.2 `recognize_pdf()` 参数错误

**问题：**
```
TypeError: OCRService.recognize_pdf() takes 2 positional arguments but 3 were given
```

**原因：** `api/ocr.py` 中调用 `ocr_service.recognize_pdf(pdf_file, pages)` 时传了两个参数，但 `recognize_pdf()` 方法只接受一个参数（`pdf_path`）。

**发现过程：** 早期 `ocr_service.py` 中 `recognize_pdf()` 方法确实有 `pages` 参数用于指定要处理的页面列表，但后续重构移除了该参数（改为处理全部页面），`api/ocr.py` 中的调用代码未同步更新。

**修复：** 移除 `api/ocr.py` 中多余的 `pages` 参数。
```python
# 修复前
ocr_result = ocr_service.recognize_pdf(pdf_file, pages)

# 修复后
ocr_result = ocr_service.recognize_pdf(pdf_file)
```

### 5.3 不同 DPI 的 PDF 无法对比

**问题：**
```
ValueError: 图像尺寸不一致: (877, 638) vs (1754, 1241)
```

**原因：** 模板 PDF 以 150 DPI 渲染后尺寸为 877×638，而扫描件 PDF 以 300 DPI 渲染后尺寸为 1754×1241（宽高均为 2 倍）。`compute_diff_mask()` 在尺寸不一致时直接抛出异常。

**分析：**
- 扫描仪默认 300 DPI，扫描件尺寸更大
- 模板 PDF 直接用 PDF 软件打开（150 DPI 左右）
- 两者的像素分辨率天然不同

**解决方案：** 在 `template_service.py` 中添加 `_normalize_image_size()` 方法：
1. 以**较小尺寸**为目标（避免放大引入人为差异）
2. 使用 **LANCZOS 插值**对较大图像进行缩小
3. 缩放后的差异更真实反映内容差异（而非 DPI 差异）

```python
def _normalize_image_size(self, img1, img2):
    h1, w1 = img1.shape[:2]
    h2, w2 = img2.shape[:2]
    if h1 == h2 and w1 == w2:
        return img1, img2
    # 以较小尺寸为目标，避免放大引入人为差异
    target_h, target_w = min(h1, h2), min(w1, w2)
    # ... resize 到 target_h, target_w
```

### 5.4 numpy 类型无法 JSON 序列化

**问题：**
```
TypeError: Object of type int64 is not JSON serializable
```

**原因：** `PaddleOCRVL` 的 `pipeline.predict()` 返回的结果对象中，bbox 坐标等数值字段使用 numpy int64/float64 类型：
- `ocr_result.raw.res.parsing_res_list[*].block_bbox` → `numpy.ndarray`
- `ocr_result.raw.res.width/height` → `numpy.int64`

这些 numpy 类型在 `json.dump()` 时会抛异常。

**排查过程：**
1. 初期以为在 `api/ocr.py` 中转换即可 → 失败（顶层转换了但嵌套层仍有）
2. 在 `api/ocr.py` 添加 `check_serializable()` 调试函数 → 发现 `_ensure_native` 没有被调用到
3. 定位到根本原因：`_ensure_native` 函数存在但没有被放在 `recognize` 方法的 return 语句之前
4. 在 `ocr_service.py` 的所有返回路径上添加 `_ensure_native()` 调用

**最终解决方案：**

1. **`ocr_service.py`** — 深度转换所有返回值：
```python
def _ensure_native(obj):
    """将 numpy 类型深度转换为 Python 原生类型"""
    import numpy as np
    if isinstance(obj, np.integer):
        return int(obj)
    if isinstance(obj, np.floating):
        return float(obj)
    if isinstance(obj, np.ndarray):
        return _ensure_native(obj.tolist())
    if isinstance(obj, dict):
        return {k: _ensure_native(v) for k, v in obj.items()}
    if isinstance(obj, (list, tuple)):
        return [_ensure_native(v) for v in obj]
    return obj
```

2. **`api/ocr.py`** — `save_results()` 使用 `default` 参数兜底：
```python
def save_results(task_id, results):
    def json_default(obj):
        if isinstance(obj, np.integer): return int(obj)
        if isinstance(obj, np.floating): return float(obj)
        if isinstance(obj, np.ndarray): return obj.tolist()
        raise TypeError(f"Object of type {type(obj)} is not JSON serializable")

    with open(result_file, "w", encoding="utf-8") as f:
        json.dump(results, f, ensure_ascii=False, indent=2, default=json_default)
```

### 5.5 性能问题：OCR 速度过慢

**问题：** 使用 PaddleOCRVL 1.5 进行全页 OCR，单页处理时间约 114 秒（CPU 模式），21 页文档需要约 40 分钟。

**原因分析：**
- PaddleOCRVL 是视觉-语言模型，参数规模大（1.5B 参数）
- CPU 模式下推理速度远慢于 GPU
- 报告表有大量固定格式（表格线、说明文字），重复识别浪费算力

**解决方案：模板对比优化**

核心思路：报告表的模板（空白表格）是固定的，只需要识别手写内容。

处理流程：
1. 将模板 PDF 和扫描件 PDF 转为灰度图像
2. 逐像素差分，找出所有差异区域（手写内容）
3. 仅对差异区域调用 OCR
4. 差异区域面积通常只占页面的 5-10%

**性能收益：**

| 模式 | 处理范围 | OCR 调用次数 | 预估耗时（21页） |
|------|----------|-------------|----------------|
| 普通模式 | 整页 OCR | 21 次/页 | ~40 分钟/文档 |
| **模板对比** | 仅差异区域 | ~20 区域/页 | **~10 分钟/文档** |

**已验证测试结果（2026-04-14）：**
- 差异区域检测：20 个（占页面 4.51% 面积）
- OCR 成功识别出内容，如 `"日期 ___ 年 ___ 月 ___ 日\n人"`
- 全流程正常完成，无错误
